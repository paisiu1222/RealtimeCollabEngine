📅 总体开发周期：8周 (约2个月)
工作模式：单人开发，每周40小时 技术栈：C++14 + Boost.Asio + SQLite3 + Crow + nlohmann/json

🚀 阶段一：基础设施搭建 (Week 1-2)
目标：完成项目骨架、网络通信基础及数据持久化层。

Week 1: 项目初始化与核心依赖
Day 1-2: 建立 CMake 构建系统，集成 Boost.Asio, SQLite3, Crow 等库，设计目录结构。
Day 3-4: 实现线程安全的 Logger 和基于 JSON 的 Config 解析器。
Day 5-7: 设计数据库 Schema，实现 Database 连接池及 DocumentDAO。
Week 2: 网络层基础
Day 1-3: 基于 Boost.Asio 实现 WebSocketServer，处理长连接与心跳机制。
Day 4-5: 集成 Crow 框架，搭建 HTTP API 基础路由（文档管理、用户认证）。
Day 6-7: 联调测试，确保 WebSocket 消息收发正常，进行初步的压力测试（100并发）。
Milestone 1 验收：项目可编译运行，WebSocket 连接稳定，基础 API 可用，单元测试覆盖率 > 60%。

⚙️ 阶段二：核心同步引擎 (Week 3-4)
目标：实现 OT 算法，完成多人实时协作的核心逻辑。

Week 3: OT 算法实现
Day 1-2: 定义 Operation 结构和 DocumentState 状态机。
Day 3-5: 编写 OTAlgorithm 核心转换逻辑（INSERT-INSERT, INSERT-DELETE, DELETE-DELETE）。
Day 6-7: 实现操作应用逻辑 applyOperation，确保版本号递增和历史记录管理。
Week 4: 同步协议与广播
Day 1-3: 定义 WebSocket 消息协议（JSON 格式），实现序列化与 ACK 确认机制。
Day 4-5: 实现 Room 房间管理，支持用户加入/离开及操作广播。
Day 6-7: 端到端测试，模拟双用户同时编辑，验证冲突自动解决能力。
Milestone 2 验收：OT 算法正确，多用户协作演示成功，P95 延迟 < 100ms，吞吐量 > 500 ops/sec。

🛡️ 阶段三：高级功能与优化 (Week 5-6)
目标：增强系统的健壮性，支持离线编辑和数据恢复。

Week 5: 冲突解决与离线支持
Day 1-2: 完善 ConflictResolver，实现版本冲突检测。
Day 3-4: 实现“最后写入获胜”的自动合并策略。
Day 5-7: 开发离线编辑队列 OfflineQueue，支持断线重连后的批量同步。
Week 6: 持久化与快照恢复
Day 1-3: 实现 SnapshotManager，每 100 次操作生成一次文档快照。
Day 4-5: 优化数据库写入性能（批量插入、异步刷盘）。
Day 6-7: 实现从“最新快照 + 增量日志”恢复文档的逻辑。
Milestone 3 验收：离线编辑能正确同步，系统重启后数据不丢失，快照加载速度快。

🏁 阶段四：监控、测试与交付 (Week 7-8)
目标：完善监控系统，进行全面测试并整理交付物。

Week 7: 监控与安全性
Day 1-3: 实现 Metrics 模块，统计活跃连接数、操作吞吐量。
Day 4-5: 增加 JWT 简化版认证，实现速率限制（100 ops/min）。
Day 6-7: 完善错误处理与日志分级（INFO, WARNING, ERROR）。
Week 8: 最终测试与文档
Day 1-3: 全量回归测试，修复已知 Bug，进行内存泄漏检查。
Day 4-5: 编写详细的 API 参考文档、部署指南和架构说明。
Day 6-7: 准备演示 Demo，整理代码仓库，完成项目交付。
Milestone 4 验收：所有功能符合需求文档，代码规范，文档齐全，通过最终验收标准。

💡 建议的开发优先级
先跑通流程：优先实现最简单的 INSERT 操作和单房间广播，确保链路通畅。
再攻克算法：集中精力调试 OT 算法的边界条件，这是项目的核心难点。
最后做优化：在功能完备的基础上，再进行数据库索引优化和内存池管理。
